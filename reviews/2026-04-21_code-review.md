## 即座に修正すべき問題

### 1. CoCh 判定のデータ長チェック不足 [高]
**箇所**: main.py:554-572
`detect_coch()` は `lookback*2+2 = 8` 本のバーが最低必要だが、実運用コードにデータ長の検証がない。backtest_improved.py:247 では `idx >= coch_lookback * 2 + 2` でチェック済みだが、main.py には未実装。

**影響**: データ不足時の誤検知リスク（実害は低いが防御的に修正すべき）。

### 2. レジーム判定の latest_close NaN チェック不足 [高]
**箇所**: strategy.py:290-322
`latest_sma50` / `latest_sma200` の NaN チェックはあるが、`latest_close` のチェックがない。日経225データ取得失敗時に NaN が来ると比較演算が常に False → `neutral` にフォールバック → `bull_only_entry=true` の場合は全エントリー停止。

**修正案**: `latest_close` も NaN チェックを追加し、データ不足時は `"unknown"` を返す。

### 3. valuation ストップの一方向調整 [中]
**箇所**: main.py:609-624
`tight_stop = current * 0.97` で固定設定するため、株価急落時にストップが緩む（上がる）可能性。

**修正案**: `tight_stop = max(tight_stop, pos["stop_price"])` でストップが緩まないように制御。

## 運用で回避可能な問題

### 4. イグジット価格の1日遅延 [高]
**箇所**: main.py:595
寄り前スキャン（8:50）では `df["Close"].iloc[-1]` が前営業日終値 → ストップ割れ検知が1日遅延。

**現状の影響**: 4/20 の3銘柄損切りで約 ¥630 の追加損失。

**対策**: 引け後スキャン（15:10）でのみイグジット判定を行う運用ルール、またはリアルタイム価格 API の導入。

### 5. trades.json の競合制御 [中]
**箇所**: portfolio.py:25-47
読み込み→変更→書き込みの間にロックが切れる（共有ロックのみ）。cron と MCP サーバーの同時実行でデータ破損リスク。

**対策**: 排他ロック + read-modify-write の一括処理。
